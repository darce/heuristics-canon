# Security Heuristics Lexicon

## About this document

Decision rules for security as a spanning concern: the trust boundaries,
injection sinks, privilege grants, and misconfigurations that turn a working
feature into an exploitable one. Referenced, not read end to end: a threat
model, diff, or audit cites a rule by ID where the decision happens, and this
file holds the claim behind it. Like accessibility, security cuts across the
other lexicons (an SQL-injection rule is also a data rule, a least-privilege
rule is also an architecture rule), so it earns its own lexicon rather than
scattering.

Scope: AI/ML and agent security, general web application security,
secure-design principles, and the PHP / WordPress / PostgreSQL stack. It does
not yet cover cryptography, network security, OS hardening, or identity
protocols in depth; those await both the sources and the pass.

| WEB-29<a name="web-29"></a> | Session identifiers are sequential, structured, or otherwise low-entropy | **Unpredictable session tokens** тАФ generate high-entropy, meaningless session IDs, or predictable tokens let an attacker mint another user's session | Given a sample of session IDs, can the next be forecast or mapped to a user? | B┬╖w | [stuttard-wahh](#src-stuttard-wahh) |

| WEB-33<a name="web-33"></a> | Authorization checks are copy-pasted per handler; some paths are unchecked | **Deny-by-default authorization on every path** тАФ enforce access control through shared, session-driven, deny-by-default checks covering every sensitive handler (one module or equivalent middleware), or piecemeal per-handler checks miss paths and rot under change | Is there any code path that serves sensitive data without the shared deny-by-default authz check? | S┬╖p | [stuttard-wahh](#src-stuttard-wahh) |
| WEB-34<a name="web-34"></a> | A high-impact action (add payee, transfer, role change) proceeds on the ambient session alone | **Re-authenticate for critical actions** тАФ require fresh reauth or dual authorization before an irreversible or high-value mutation, so a stolen session or CSRF cannot complete it | Can a critical mutation complete with only the existing session cookie? | S┬╖w | [stuttard-wahh](#src-stuttard-wahh) |

## 7. Cross-lexicon links

Security is a spanning concern, so many rows restate a rule another lexicon owns from the security angle:

- **Injection is one mechanism across three sinks.** SQL parameterization [[WEB-01]](security.md#web-01)/[[PHP-15]](security.md#php-15)/[[WP-01]](security.md#wp-01)/[[PG-07]](security.md#pg-07) is engineering's [[DATA-12]](engineering.md#data-12) read as an attack; the LLM analogue is [[SEC-02]](security.md#sec-02) (treat model output as untrusted before any interpreter).
- **Least privilege repeats up the stack.** [[PG-01]](security.md#pg-01) (non-superuser app role) and [[SEC-04]](security.md#sec-04) (least-privilege agency) are the same rule at the database and the model; both bound the blast radius of a compromised component (тЖФ eng [[ARCH-02]](engineering.md#arch-02), single writer owns the table).
- **Secrets discipline is one rule.** [[WEB-16]](security.md#web-16)/[[PG-09]](security.md#pg-09) and [[SEC-06]](security.md#sec-06) all say the same thing: anything a component can read, an attacker who reaches that component can read too.
- **Rate limiting is denial-of-wallet and DoS at once.** [[WEB-17]](security.md#web-17) and [[SEC-08]](security.md#sec-08) share a source mechanism; unmetered access is both an availability and a cost failure.
- **Validation at the trust boundary** ([[SEC-01]](security.md#sec-01), [[PHP-08]](security.md#php-08)) is the security face of the same boundary the engineering lexicon guards for correctness ([[API-11]](engineering.md#api-11) round-trip, [[DATA-03]](engineering.md#data-03) schema compat).
