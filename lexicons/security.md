# Security Heuristics Lexicon

## About this document

Decision rules for security as a spanning concern: the trust boundaries,
injection sinks, privilege grants, and misconfigurations that turn a working
feature into an exploitable one. Referenced, not read end to end: a threat
model, diff, or audit cites a rule by ID where the decision happens, and this
file holds the claim behind it. Like accessibility, security cuts across the
other lexicons (an SQL-injection rule is also a data rule, a least-privilege
rule is also an architecture rule), so it earns its own lexicon rather than
scattering.

Scope: AI/ML and agent security, general web application security,
secure-design principles, and the PHP / WordPress / PostgreSQL stack. It does
not yet cover cryptography, network security, OS hardening, or identity
protocols in depth; those await both the sources and the pass.

| SEC-17<a name="sec-17"></a> | A capture-facing identity system reports only identity match rates (true-accept, false-match) with no attack-presentation error table | **Attack rates are not match rates**: when live captures drive identity decisions, report presentation-attack detection error separately from identity error (artefacts accepted as genuine, and genuine presentations rejected) at a declared detection threshold and per named attack class, because a true-accept rate is measured against cooperative subjects and says nothing about how often a print, replay, or mask is accepted (↔ [[SEC-14]](security.md#sec-14) the same clean-metric blindness, digital perturbation rather than physical presentation) | Are attack-presentation error rates reported separately from identity false-match rates, at a declared detection threshold? | S·r | [handbook-face-recognition](../SOURCES.md#src-handbook-face-recognition) |
| SEC-18<a name="sec-18"></a> | An accept fires on similarity score while the liveness check runs after the accept, is logged only, or is fused with no written policy | **The liveness verdict must gate the accept**: when a live capture can be a print, replay, or worn mask, the presentation-attack stage must reject before the gallery comparison commits an accept; if the two scores are fused instead, the fusion policy and both thresholds must be written down and testable, because an attacker holding an enrolled person's photograph scores as that person, and a liveness signal nothing blocks on is telemetry, not a control | Can a presented photograph or replayed video of an enrolled person reach a final accept on similarity alone? | B·w | [handbook-face-recognition](../SOURCES.md#src-handbook-face-recognition) |
| SEC-19<a name="sec-19"></a> | Liveness is an interactive motion challenge (blink, smile, turn head) and the attack evaluation set holds only flat printed photographs | **A motion challenge is not an attack set**: when liveness rests on interactive motion, extend the evaluation set to replayed video and to partial or three-dimensional artefacts (cut-out regions, worn masks, attack eyewear) before the ship gate passes, because a recording of the enrolled person performing the requested action satisfies a motion challenge, and a region-specific check is defeated by an artefact covering only that region | Does the attack set include replayed video and partial or three-dimensional artefacts, or only static print? | S·p | [handbook-face-recognition](../SOURCES.md#src-handbook-face-recognition) |
| SEC-20<a name="sec-20"></a> | Presentation-attack detection is certified on the same camera, lighting, and capture protocol it was trained on, while the product ships to varied client devices | **Certify liveness on held-out capture hardware**: when client sensors, resolution, or lighting differ from the training capture, report detection error under a leave-one-device or leave-one-capture-source protocol, because cross-source error degrades sharply under illumination, resolution, and sensor-noise shift, so a matched-domain number certifies the lab and the first unseen camera is a field bypass rather than a regression (↔ ml [[FAIR-05]](ml-systems.md#fair-05) capture conditions are evaluation factors: this is the held-out form of that factor) | Is there a leave-one-device table, or only same-camera lab numbers? | S·r | [handbook-face-recognition](../SOURCES.md#src-handbook-face-recognition) |

## 2. Web Application Security<a name="fam-web"></a>

> Re-grounded against Zalewski, *The Tangled Web* (`zalewski-tangled-web`, browser security model) and Stuttard & Pinto, *The Web Application Hacker's Handbook* (`stuttard-wahh`, server-side attacks). The four rows those texts do not establish ([[WEB-12]](security.md#web-12), [[WEB-16]](security.md#web-16), [[WEB-19]](security.md#web-19), [[WEB-20]](security.md#web-20)) remain `bootstrap`.

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| WEB-01<a name="web-01"></a> | User-controlled data concatenated into an SQL string (or interpolated into a query without a binder) | **Parameterize every query**: string-built SQL lets any quote/terminator break out into code; use prepared statements + bound params, including for values re-read from the database (second-order injection); binders bind values, so dynamic identifiers need [[WEB-35]](security.md#web-35) (↔ eng [[DATA-12]](engineering.md#data-12) both fail when a value is interpolated into a query instead of bound) | Is every SQL value bound, including stored values re-used in a later query? | B·w | [stuttard-wahh](../SOURCES.md#src-stuttard-wahh) |
| WEB-02<a name="web-02"></a> | User input passed to `exec`/`system`/`shell_exec`/`Runtime.exec`/`ProcessBuilder` without an argv array | **Never shell-interpolate untrusted input**: metacharacters become a second command; use argv APIs or refuse shell (↔ eng [[DATA-12]](engineering.md#data-12) never interpolate untrusted bytes into an interpreter) | Is the OS command built as discrete args with no shell? | B·w | [stuttard-wahh](../SOURCES.md#src-stuttard-wahh) |
| WEB-03<a name="web-03"></a> | LDAP filter built with string concat of request fields | **Allowlist LDAP filter input**: `)(uid=*` style input widens the filter and bypasses authz; permit only simple allowlisted characters and reject the filter metacharacters (parentheses, `*`, `&`, `=`, pipe) and null bytes, or use a typed directory API (WAHH's stance is reject, not sanitize-in-place) (↔ eng [[DATA-12]](engineering.md#data-12) metacharacter breakout from data into control plane) | Is every LDAP filter value allowlisted and metacharacter-rejected? | B·w | [stuttard-wahh](../SOURCES.md#src-stuttard-wahh) |
| WEB-04<a name="web-04"></a> | Response body echoes request data (or stored user content) into HTML without context-aware encoding | **Encode for the sink context**: unescaped HTML lets scripts run as the victim (reflected/stored XSS) | Is every untrusted string escaped for HTML/attr/JS/URL before emit? | B·w | [zalewski-tangled-web](../SOURCES.md#src-zalewski-tangled-web) |
| WEB-05<a name="web-05"></a> | Client-side code writes `location`/`document.write`/`innerHTML` from `location.hash`/`postMessage`/URL params | **Treat DOM sinks as XSS sinks**: client-only flows still execute attacker script in the origin | Is untrusted data never assigned to HTML/JS sinks without sanitization? | B·w | [zalewski-tangled-web](../SOURCES.md#src-zalewski-tangled-web) |

| WEB-08<a name="web-08"></a> | Object ID / path param used in a query without comparing ownership or tenant to the authenticated principal | **Authorize the object, not just the route**: knowing another ID is enough for IDOR/data disclosure (↔ eng [[ARCH-13]](engineering.md#arch-13) a property left to handler vigilance is not enforced; ↔ biz [[CLM-03]](business-marketing.md#clm-03) a policy sentence the implementation does not hold; ↔ ml [[PROV-05]](ml-systems.md#prov-05) a prediction an undeclared consumer will treat as a decision; ↔ eng [[TEAM-01]](engineering.md#team-01) an architectural split the org does not communicate across; ↔ eng [[NAME-03]](engineering.md#name-03) a name that promises a type or behavior the code does not hold) | Does the handler enforce ownership/tenant before read or write? | B·w | [stuttard-wahh](../SOURCES.md#src-stuttard-wahh) |

| PG-03<a name="pg-03"></a> | `SECURITY DEFINER` function without a pinned `search_path` (or a writable schema ahead in the path) | **Pin search_path on DEFINER**: attackers plant shadow objects and hijack privileged functions | Do DEFINER functions set a fixed safe `search_path` and a trusted owner? | B·w | [postgresql-security §search-path](../SOURCES.md#src-postgresql-security) |
| PG-04<a name="pg-04"></a> | `pg_hba.conf` uses `trust`/`md5` for remote or broad CIDR matches | **Prefer scram-sha-256 auth**: `trust` is passwordless access; md5 is weaker than SCRAM | Are remote entries scram-sha-256 (not trust/md5) with tight CIDRs? | B·r | [postgresql-security §auth-hba](../SOURCES.md#src-postgresql-security) |
| PG-05<a name="pg-05"></a> | Connections allowed without SSL, or `sslmode` defaults to prefer/disable in app DSNs | **Require TLS to Postgres**: cleartext credentials and rows are MitM-visible | Is server SSL on and are clients using verify-full/verify-ca as appropriate? | B·r | [postgresql-security §ssl-tls](../SOURCES.md#src-postgresql-security) |
| PG-06<a name="pg-06"></a> | `GRANT … TO PUBLIC`, or default public-schema privileges left world-usable | **Revoke PUBLIC grants**: PUBLIC privileges give every role unintended DML/EXECUTE | Are unnecessary PUBLIC grants revoked on schemas, tables, and functions? | S·r | [postgresql-security §privileges](../SOURCES.md#src-postgresql-security) |
| PG-07<a name="pg-07"></a> | PL/pgSQL or SQL builds dynamic statements with string concat of identifiers/literals | **Use format(%I/%L) or EXECUTE … USING**: dynamic SQL at the DB layer is still SQLi (↔ [[WEB-01]](security.md#web-01) concat of identifiers/literals is injection even inside PL/pgSQL) | Is every dynamic identifier/literal passed via `%I`/`%L` or a bind param? | B·w | [postgresql-security §dynamic-sql](../SOURCES.md#src-postgresql-security) |
| PG-08<a name="pg-08"></a> | Untrusted users can `CREATE EXTENSION`, or extensions installed cluster-wide without review | **Treat extensions as code**: unpackaged C/SQL extensions expand attack surface and privilege | Are only required extensions installed, by a superuser, after review? | S·r | [postgresql-security §extensions](../SOURCES.md#src-postgresql-security) |
| PG-09<a name="pg-09"></a> | DB passwords in app repos, a world-readable `.pgpass`, or a shared superuser password across envs | **Isolate DB secrets**: keep passwords out of VCS and world-readable files; mode `0600` on `.pgpass`; prefer a secret store in app deploy; a leaked password equals full data access for that role (↔ [[WEB-16]](security.md#web-16) credentials outside a controlled store are permanent credential theft) | Are DB credentials kept out of VCS and world-readable files, with `.pgpass` mode 0600? | B·r | [postgresql-security §secrets](../SOURCES.md#src-postgresql-security) |
| PG-10<a name="pg-10"></a> | No base backups + WAL archiving, or an untested PITR restore path | **Prove backups restore**: untested backups fail when ransomware or a `DROP` happens | Is PITR configured and a restore drill documented as successful? | B·p | [postgresql-security §backup-pitr](../SOURCES.md#src-postgresql-security) |
| PG-11<a name="pg-11"></a> | `log_statement` off for privileged and DDL activity; no durable log retention | **Log privileged SQL**: set at least `log_statement = ddl` (or finer) and retain logs via `logging_collector`; without a trail, abuse and SQLi exfiltration go unnoticed (`pgaudit` is an optional extension, not core) (↔ ux [[HAI-06]](interaction-ux.md#hai-06) without a durable log, abuse cannot be reconstructed) | Are DDL/role changes and sensitive statements logged and retained? | S·r | [postgresql-security §logging](../SOURCES.md#src-postgresql-security) |
| PG-12<a name="pg-12"></a> | `max_connections` high with no per-role `CONNECTION LIMIT`; app opens unbounded pools | **Cap connections**: set `max_connections` and per-role `CONNECTION LIMIT` below collapse thresholds; connection storms exhaust the postmaster (an external pooler is operational, not a PostgreSQL-doc requirement) (↔ eng [[RES-14]](engineering.md#res-14) both fail when an unbounded producer exhausts a fixed resource pool) | Are server and per-role connection limits set below collapse thresholds? | S·r | [postgresql-security §connections](../SOURCES.md#src-postgresql-security) |
| PG-13<a name="pg-13"></a> | Views/functions expose rows that RLS on base tables would hide (`security_barrier`/owner ignored) | **Align views with RLS**: a non-barrier view can leak filtered rows | Do views and DEFINER paths preserve the intended row filters? | S·w | [postgresql-security §security-barrier](../SOURCES.md#src-postgresql-security) |
| PG-14<a name="pg-14"></a> | Replication or backup role holds broad write privileges beyond need | **Separate backup/repl roles**: an over-privileged ops role expands blast radius if stolen | Do backup/replication roles have only the minimal REPLICATION/read grants? | S·r | [postgresql-security §replication-roles](../SOURCES.md#src-postgresql-security) |

## 6. Secure Design Principles<a name="fam-secd"></a>

> The cross-cutting engineering principles beneath every section above, from Ross Anderson, *Security Engineering*, 3rd ed. (`anderson-security-engineering`). Where a principle is the general form of a specific rule (least privilege → [[SEC-04]](security.md#sec-04)/[[PG-01]](security.md#pg-01)/[[SEC-16]](security.md#sec-16); complete mediation → [[WEB-31]](security.md#web-31)/[[WEB-33]](security.md#web-33)), the specific rule cross-links here rather than restating it. See [Principle 14].

| SECD-01<a name="secd-01"></a> | Any system carries a protection claim ("secure", "safe", "compliant") | **Write the security policy first**: state in one paragraph what must not happen and who may do what, before choosing mechanisms; mechanism without a falsifiable policy is theatre (↔ biz [[PROD-09]](business-marketing.md#prod-09) write the channel contract before the components; ↔ a11y [[A11Y-22]](accessibility.md#a11y-22) "secure" is a falsifiable sentence or it is empty; ↔ biz [[CLM-05]](business-marketing.md#clm-05) three hostile readers who check every claim against a written spec; ↔ eng [[RLSE-09]](engineering.md#rlse-09) expert sign-off is exact quoted strings and yes/no questions, not jargon that shifts liability) | Can an engineer falsify compliance with the policy in a test? | S·p | [anderson-security-engineering](../SOURCES.md#src-anderson-security-engineering) |
| SECD-02<a name="secd-02"></a> | A component whose compromise would break the policy | **Minimize the trusted computing base**: treat it as **trusted** and shrink that set: split duties, sandbox untrusted code, remove unneeded keys and scopes, because a fat TCB makes every bug fatal (↔ [[SEC-04]](security.md#sec-04) a jailbreak exercises every scope it holds; ↔ [[PG-01]](security.md#pg-01) app SQLi is cluster takeover if the role is superuser; ↔ [[SEC-16]](security.md#sec-16) one identity that can silently swap production weights; ↔ [[PG-06]](security.md#pg-06) PUBLIC grants hand every role unintended DML; ↔ [[PG-14]](security.md#pg-14) an over-privileged ops role expands the blast radius of a stolen credential; ↔ [Principle 14]) (↔ [[SECD-03]](security.md#secd-03) least privilege shrinks what a grant reaches; complete mediation checks every use of it, the two halves of one blast-radius discipline) | If this principal is malicious, is the whole policy dead? | B·p | [anderson-security-engineering](../SOURCES.md#src-anderson-security-engineering) |

## 7. Cross-lexicon links

Security is a spanning concern, so many rows restate a rule another lexicon owns from the security angle:

- **Injection is one mechanism across three sinks.** SQL parameterization [[WEB-01]](security.md#web-01)/[[PHP-15]](security.md#php-15)/[[WP-01]](security.md#wp-01)/[[PG-07]](security.md#pg-07) is engineering's [[DATA-12]](engineering.md#data-12) read as an attack; the LLM analogue is [[SEC-02]](security.md#sec-02) (treat model output as untrusted before any interpreter).
- **Least privilege repeats up the stack.** [[PG-01]](security.md#pg-01) (non-superuser app role) and [[SEC-04]](security.md#sec-04) (least-privilege agency) are the same rule at the database and the model; both bound the blast radius of a compromised component (↔ eng [[ARCH-02]](engineering.md#arch-02), single writer owns the table).
- **Secrets discipline is one rule.** [[WEB-16]](security.md#web-16)/[[PG-09]](security.md#pg-09) and [[SEC-06]](security.md#sec-06) all say the same thing: anything a component can read, an attacker who reaches that component can read too.
- **Rate limiting is denial-of-wallet and DoS at once.** [[WEB-17]](security.md#web-17) and [[SEC-08]](security.md#sec-08) share a source mechanism; unmetered access is both an availability and a cost failure.
- **Validation at the trust boundary** ([[SEC-01]](security.md#sec-01), [[PHP-08]](security.md#php-08)) is the security face of the same boundary the engineering lexicon guards for correctness ([[API-11]](engineering.md#api-11) round-trip, [[DATA-03]](engineering.md#data-03) schema compat).
